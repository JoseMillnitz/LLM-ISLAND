# Scenario — Cross-Language Pipeline
# Read this only when the project uses more than one programming language,
# or has files that compile/transpile into other formats.

---

## SITUATION

The project spans multiple languages: Python calling C, C compiling to Assembly,
TypeScript transpiling to JavaScript, a build pipeline with generated files, etc.

---

## CORE RULE

One island per source file, regardless of language.
Each file is described in the terms of its own language.
Neither island needs to understand the other language.
The mainland holds the translation contract between them.

---

## PROCESS

Step 1 — Identify all language boundaries in the project.
  A language boundary is any point where one language calls into another,
  or where a file is produced by transformation of another file.

Step 2 — Assign `layer: bridge` to files that mediate between languages.
  Bridge files explicitly describe both sides of the boundary they serve.

Step 3 — Declare each boundary as a CONNECTION in the mainland with
  the `language-boundary` field:

```
CONNECTION: orchestrator.py -> renderer.c
  uses:              render_frame(), init_context()
  why:               Python delegates frame rendering to C for performance
  direction:         bridge -> core
  language-boundary: python -> c via ctypes
  strength:          critical
  break-impact:      all rendering fails
```

Step 4 — Create islands for both files at the boundary.
  The Python island describes what it calls and what it expects in Python terms.
  The C island describes what it exports and what it expects from callers in C terms.
  Neither references the other language's syntax or semantics.

---

## WHEN THE IMPLEMENTATION CHANGES BUT THE CONTRACT DOES NOT

If the C renderer is rewritten in Assembly:
  1. Update or replace the C island with an Assembly island
  2. Update the mainland connection: `language-boundary: python -> asm via nasm`
  3. The CONTRACT governing boundary behavior stays unchanged
  4. The considerations log records the rewrite as a HISTORICAL-DECISION:
     "renderer.c rewritten to renderer.asm, interface preserved"

The semantic contract at the boundary outlives the implementation language.
This is the key property of cross-language islands.

---

## GENERATED FILES

If a file is generated by a build step (e.g. compiled output, transpiled JS,
generated protobuf code):
  - The source file gets a normal island
  - The generated file gets an island marked `status: generated-artifact`
  - The mainland connection declares: `generated-by: <build-step>`
  - The generated file's island does not need MEMORY — it has no decisions,
    only the decisions of its source file

---

## KEY RULE

The mainland connection is the single source of truth about what crosses a
language boundary. If the boundary breaks, the mainland connection is where
you look first — not the individual islands.
